How to Arch Linux

26 February 2017

Preparation

First, prepare yourself. Think and answer:

  • Are you trying to be cool/edgy?
  • Are you too pragmatic to live the FOSS life?
  • Does the term "CLI" foster warm fuzzy feelings?
  • Does not knowing what bootloader your system uses discomfort you?
  • Do you get mad when the new version your operating system decides to send your search results to Amazon, so you give up and switch to Debian but get irked by obsolete documentation and package updates that move so slowly that you often need to download source, compile, then link back to your package manager (after removing any conflicts), so often that you've learned all the error messages that dpkg -i can throw at you?
Just kidding about the last one. Sort of. But do be ready to learn a lot through trial and error, and reading the Arch Wiki.

You will need:

  • Computer (bigger is better)
  • Internet (wired is easier)
  • Bootable media (flash drives are easy)
  • Reliable access to the Arch Wiki

Prepare your installation media. Download Arch Linux and dd. If you've forgotten how to dd, perhaps this command will help: 

dd if=archlinux-2017.02.01-dual.iso of=/dev/sdb
Don't forget to lsblk.
If you don't know what you're doing, consult the Arch Wiki.

Installing

I'm going to assume you've booted into Arch Linux now from your bootable media.

If you have an Ethernet connection, your internet should be working already. Otherwise, run 

wifi-menu
to connect to your wireless network.

Update your system time. 

timedatectl set-ntp true
If your system time is off, bad things happen. Like SSL/TLS certificates not working.

Hopefully you know which disk you want to install Arch Linux onto. If not, run lsblk and be extra sure. Partition your disks. 

cfdisk
It should be fast and easy. If you don't know why you're partitioning the disks, just make a big, bootable partition:

Now format your partition(s). If using an SSD, f2fs is nice for longevity. ReiserFS is nice for your log files on /var. If you're not doing anything remarkable, then ext4 will be good: 

mkfs.ext4 /dev/sda1

Mount the file system. /mnt is the standard mount point. 

mount /dev/sda1 /mnt
If you had multiple partitions, then you will need to make the directories on which to mount the partitions.

Now that your disks are prepared, it is time to add Arch Linux files. Choose where you would like to download from. 

vim /etc/pacman.d/mirrorlist
If you don't know how to use vim, press the following keys after opening the file: 
/StatesEnterjddggP:qEnter
Not skipping this step and choosing a mirror makes the following download significantly faster: 
pacstrap /mnt base
This downloads Arch Linux onto your disks.

Now you want to save the configuration of your mount points: 

genfstab -U /mnt >> /mnt/etc/fstab

After this step, you have a working system: 

arch-chroot /mnt
How exciting! If you want this system to be loaded at boot however, you will need to install a bootloader. GRUB is easy and supports nearly everything. Assuming you've installed Arch Linux to /dev/sda, then you need only run: 
pacman -S grub
grub-install --target=i386-pc /dev/sda
grub-mkconfig -o /boot/grub/grub.cfg
These three commands install grub through the package manager and configure it for your /dev/sda device.

Now is a good time to figure out your wireless software, if you need that. Setting a locale is also important, so running 

echo "en_US.UTF-8 UTF-8" >> /etc/locale.gen && locale-gen
echo "LANG=en_US.UTF-8" >> /etc/locale.conf
is useful for English locales.

That's it! reboot and realize that cross-platform is a lie! Go figure out what went wrong, unless you're one of the lucky few with Linux-friendly hardware.

If you're stuck, Anthony Weems has a really great copypasta for a key-by-key minimalist Arch installation (using syslinux and minimal configurations). Mirrored here.

Did I mention that the Arch Wiki is excellent ?

Go Learn a New Language

18 February 2017

A common language ties communities together, while language differences incite distrust and misunderstanding. A minority of people speak only one language , and far fewer programmers claim proficiency with only one programming language. If you don't consider yourself a linguanaut, go do yourself a favour and explore something new.

Choosing a programming language is a first step for any aspiring programmer. After learning about the diversity of ways in which computers may be communicated with, a thought comes about: "Which language is the best?" Unless you spend a lot of time with computers, and a good proportion of that time is spent programming, your first programming language will be a significant investment in time and effort. For self-taught learners with the luxury (and burden) of choice, it's time to do some research:

The language debate can get quite heated online. Caricatures abound. Eventually the sorts of projects you want to work on will define what tools are most appropriate, and your first language will often not be your strongest -- it's more important for a first-time swimmer to get in water than to choose a preferred stroke. Pick something accessible and up to modern standards (not COBOL, not BASIC, not VBScript). Anything else and it won't make a big difference, really.

Programming ecosystems are diverse, and programmers even more so. Sooner or later, every programmer picks up some bash, maybe some shader code, and a formal university education will put you through some C and a scripting language. After studying a few languages, you should see that it becomes much easier to learn related languages. Java programmers should have little trouble picking up Kotlin, and Ruby programmers should find Python quite accessible. It's important to keep learning languages that are 'relatively hard' such as pure functional languages if you come from a declarative background, or assembly languages if you come from a high-level background. The world needs more Erlang programmers, and good CUDA programmers are hard to come by.

.

Think -- if the West was educated in Modern Standard Arabic (the 5th most used language today), would oxymoronic phrases such as "Islamic Terrorism" ever resound with the public? When Python programmers know the difference between built-in arrays and numpy arrays, when C++ programmers see lambda expressions for what they are, when C programmers can see times when Haskell would be more appropriate, then maybe we can make Stack Overflow great again.
